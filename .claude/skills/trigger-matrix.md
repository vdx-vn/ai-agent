# Trigger Matrix

Use this matrix to keep routing precise. `Must trigger` means the skill is the primary owner. `Must not trigger` marks overlap boundaries. `Tie-breaker` gives the hard routing rule when neighboring skills share vocabulary.

| Skill | Class | Primary artifact | Must trigger on | Tie-breaker | Must not trigger on |
|---|---|---|---|---|---|
| `odoo-think` | task | Risk-oriented scoping brief with module map, business entrypoint, cross-app impact, and open questions. | scope request, impact analysis, what modules affected, what could break | Use this skill only when the primary requested output is a scoping brief or impact diagnosis. If the user wants ordered steps, file-by-file actions, or acceptance criteria, hand off to `odoo-plan`. | implementation plan, code review, test execution |
| `odoo-plan` | task | Ordered execution plan with file map, acceptance criteria, test matrix, rollout notes, and open decisions. | implementation plan, step-by-step approach, what files to change, acceptance criteria | Use this skill only when the user wants a plan artifact. If the user wants a risk brief, use `odoo-think`. If the user wants code edits, use `odoo-build`. | pure scoping, coding, retrospective |
| `odoo-build` | task | Concrete in-scope code or configuration changes plus noted assumptions and follow-up validation needs. | implement this, write Odoo code, add field/view/action, modify model logic | Use this skill only when the primary requested output is changed code or config artifacts. If the user is still deciding approach, use `odoo-plan`. | planning only, review only, pure testing guidance |
| `odoo-review` | task | Structured findings on an existing diff or artifact, separated into required fixes, open risks, and optional improvements. | review this patch, code review, inspect diff, pre-merge review | Use this skill only when the primary requested output is review findings on an existing artifact. If the user wants executed validation evidence, use `odoo-test`. If the prompt centers on exposure or trust boundaries, compose with or defer to `odoo-security`. | test execution only, planning from scratch, retrospective |
| `odoo-test` | task | Current-change validation evidence or validation plan tied to a specific diff, addon, bug, or runtime scenario. | test this, run tests, validate module, regression test | Use this skill only when the primary requested output is validation for a concrete change now. If the user asks which Odoo test framework or tags to use in general, hand off to `odoo-testing-reference`. | abstract testing reference only, code review only, deployment orchestration only |
| `odoo-ship` | task | Go or no-go rollout checklist with staging verification, release sequencing, rollback cautions, and production notes. | ship this, release this, prepare deployment, production readiness | Use this skill only when the user wants release readiness or rollout sequencing. If the user needs CLI flag semantics or environment mechanics, hand off to `odoo-delivery-ops`. If the primary issue is data or schema evolution, compose with `odoo-upgrade-migration`. | implementation, code review, retrospective |
| `odoo-reflect` | task | Retrospective with lessons, remaining gaps, and concrete follow-up actions. | retrospective, reflect on this work, what did we learn, follow-up gaps | Use this skill only when the primary requested output is a retrospective or lessons learned artifact after work already happened. | planning, code review in progress, test execution |
| `odoo-architecture` | reference | Architecture recommendation with target addon, dependency rationale, and bridge-module guidance. | module boundary, where should code live, dependency design, bridge module | Use this skill only when the primary requested output is an addon-placement or dependency decision. | field semantics, ACL details, business flow only |
| `odoo-orm-modeling` | reference | ORM modeling guidance with field or method recommendations and anti-pattern checks. | model design, field type, compute field, onchange, constrains, recordset | Use this skill only when the primary requested output is a model or ORM design decision. | views, ACL and record rules, business flow explanation |
| `odoo-view-ui` | reference | UI guidance with view strategy, inheritance notes, and action or menu recommendations. | form view, tree view, search view, menu, action, QWeb, OWL | Use this skill only when the primary requested output is a view, action, menu, template, or client-side UI decision. | model semantics, ACL policy, business flow only |
| `odoo-security` | reference | Security assessment with trust boundaries, permission checks, exposure risks, and required safeguards. | ACL, access rights, record rules, controller auth, CSRF, sudo, SQL injection, XSS | Use this skill only when the primary requested output is a trust-boundary or exposure assessment. If the user wants a broad code review, use `odoo-review` and compose with this skill only when security is central. | generic review without security angle, pure business flow question |
| `odoo-testing-reference` | reference | Framework and test-authoring guidance for future or current test design, without claiming current-change readiness. | test tags, TransactionCase, HttpCase, tour, JS tests, how to test Odoo | Use this skill only when the primary requested output is framework or test-authoring guidance. If the user asks whether current work is tested or safe, use `odoo-test`. | live test execution, post-merge quality judgment |
| `odoo-performance` | reference | Performance guidance with hotspot hypotheses, optimization levers, and validation ideas. | slow query, N+1, query count, indexing, profiler, performance regression | Use this skill only when the primary requested output is performance diagnosis or optimization guidance. | generic ORM semantics only, shipping or release |
| `odoo-integration-api` | reference | Integration design guidance with auth model, data flow, safety considerations, and operational constraints. | external API, XML-RPC, JSON-RPC, webhook, integration user, sync job | Use this skill only when the primary requested output is an external integration design or RPC/API decision. | internal module-only customization, UI questions |
| `odoo-upgrade-migration` | reference | Migration strategy with data-preservation plan, script notes, rollout order, and validation needs. | migration, upgrade script, schema change, field rename, noupdate, version upgrade | Use this skill only when the primary requested output is migration or upgrade strategy. If the user mainly wants command semantics, use `odoo-delivery-ops`. | routine feature build, runtime config only |
| `odoo-delivery-ops` | reference | Operational mechanics guidance covering commands, flags, environment behavior, and runtime cautions. | odoo-bin flags, workers, runtime config, Odoo.sh, deployment mechanics | Use this skill only when the primary requested output is command, flag, worker, or environment semantics. If the user asks whether a release is ready, use `odoo-ship`. | migration script authoring, generic shipping narrative |
| `odoo-business-sales` | reference | Sales process map from a backend sales entrypoint, with downstream documents, roles, and cross-app impacts. | quotation, sales order, opportunity to quote, sale_crm, sale_stock | Use this skill only when the primary business entrypoint is lead, opportunity, quotation, salesperson action, or backend sales order. If the entrypoint is page, form, cart, checkout, or portal, use `odoo-business-website-ecommerce`. | website cart and checkout entrypoints, purchase flow, generic ACL |
| `odoo-business-purchase` | reference | Purchase process map with entrypoint, downstream documents, roles, and cross-app impacts. | RFQ, purchase order, vendor bill linkage, receipt from PO, purchase_stock | Use this skill only when the primary business entrypoint is RFQ, purchase order, replenishment through buying, or vendor-side procurement. | sales flow, accounting-only treatment, technical implementation patterns |
| `odoo-business-inventory` | reference | Inventory process map from a warehouse or stock-document entrypoint, with route logic, valuation, and cross-app effects. | picking, receipt, delivery, warehouse, route, replenishment, lots, serial | Use this skill only when the primary business entrypoint is receipt, delivery, transfer, route, replenishment, or stock move. If the entrypoint is BoM, MO, work center, routing, or subcontracting, use `odoo-business-manufacturing`. | BoM and MO entrypoints, accounting-only reporting, technical ORM design |
| `odoo-business-manufacturing` | reference | Manufacturing process map from a BoM or MO entrypoint, with operations, stock effects, and downstream cross-app impacts. | BoM, MO, work center, routing, subcontracting, mrp | Use this skill only when the primary business entrypoint is BoM, MO, work center, routing, subcontracting, or production planning. If the entrypoint is warehouse documents or routes without production-control logic, use `odoo-business-inventory`. | warehouse document and route entrypoints, accounting-only questions, ACL design |
| `odoo-business-accounting` | reference | Accounting process map with finance documents, posting impacts, and cross-app implications. | invoice, vendor bill, tax, fiscal position, journal, payment, reconciliation, valuation entry | Use this skill only when the primary business entrypoint is invoice, bill, payment, tax, reconciliation, localization, or valuation impact. | non-finance business flow only, generic ACL, technical placement only |
| `odoo-business-hr` | reference | HR process map from a workforce entrypoint, with employee lifecycle touchpoints and cross-app dependencies. | employees, contracts, attendances, time off, payroll prerequisites, work entries | Use this skill only when the primary business entrypoint is employee, contract, attendance, time off, planning, or payroll prerequisite. If the entrypoint is expense claim, expense sheet, reimbursement, or posting, use `odoo-business-expenses`. | expense claim entrypoints, project billing only, technical design only |
| `odoo-business-timesheet-project-services` | reference | Service-delivery process map with task, time, billing, and cross-app impacts. | project task, timesheet, service delivery, billable time, sale_project | Use this skill only when the primary business entrypoint is project, task, timesheet, helpdesk, planning, or service delivery. | general HR contracts or payroll, expense reimbursement, technical testing framework |
| `odoo-business-expenses` | reference | Expense process map from an expense-claim entrypoint, with states, roles, accounting impacts, and related HR links. | expense, expense sheet, reimbursement, approve expenses, hr_expense | Use this skill only when the primary business entrypoint is expense claim, expense sheet, reimbursement, or expense posting. If the entrypoint is employee contract, attendance, time off, or payroll prerequisite, use `odoo-business-hr`. | employee and payroll prerequisite entrypoints, general AP/vendor bills, pure field design |
| `odoo-business-website-ecommerce` | reference | Website and ecommerce process map from a public or portal entrypoint, with customer journey, downstream sales flow, and cross-app impacts. | website shop, cart, checkout, ecommerce, website_sale, website_sale_stock, website_crm | Use this skill only when the primary entrypoint is page, form, portal, cart, checkout, or online storefront behavior. If the entrypoint is lead, quotation, salesperson action, or backend sales order, use `odoo-business-sales`. | backend sales entrypoints, backend form-view design only, technical ACL or ORM design |

- `odoo-plan` vs `odoo-build` — break tie by artifact: plan vs changed files
- `odoo-review` vs `odoo-test` — break tie by artifact: findings vs validation evidence
- `odoo-test` vs `odoo-testing-reference` — break tie by artifact: current-change validation vs framework guidance
- `odoo-ship` vs `odoo-delivery-ops` — break tie by artifact: go/no-go rollout vs command semantics
- `odoo-security` vs `odoo-review` — break tie by center of gravity: trust boundary vs broad review
- `odoo-business-sales` vs `odoo-business-website-ecommerce` — break tie by business entrypoint: backend sales vs public storefront
- `odoo-business-inventory` vs `odoo-business-manufacturing` — break tie by business entrypoint: warehouse document vs MO or BoM
- `odoo-business-hr` vs `odoo-business-expenses` — break tie by business entrypoint: workforce policy vs expense claim lifecycle
